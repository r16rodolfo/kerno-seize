Kerno Seize v6.2.1 · Modo Simplificado

Kerno Lovable‑First

O fluxo sem BMAD. Para quem quer ir direto ao Lovable com o mínimo de planejamento — e ainda assim manter segurança, estabilidade e documentação.

⚡ Tempo até o primeiro commit: ~45 minutos · PRD manual no lugar de agentes

🎯 Para quem é este guia

Para o vibe coder que quer começar agora — ideia na cabeça, sem paciência para 3 horas de planejamento com agentes de IA. Você aceita algum retrabalho futuro em troca de velocidade no início.

Este guia mantém as proteções que não podem ser puladas — GitHub com SAST, RLS no Supabase, Seção A no Lovable — mas elimina tudo que é planejamento antecipado. Você planeja à medida que constrói.

⚠️ Limite deste modo: quanto mais o produto cresce, mais difícil fica sem documentação de planejamento prévia. O BMAD não é obrigatório — mas quanto maior o escopo, mais ele vai cobrar depois. Se o seu MVP tem mais de 3 features core, considere pelo menos usar Mary + John antes de continuar.

🔀 Lovable-First ou BMAD? — Guia rápido de decisão

⚡ Use este guia (Lovable-First) se:

  • A ideia tem < 3 features core bem definidas
  • Você está só testando se a ideia funciona (spike)
  • É um projeto pessoal sem usuários reais
  • Você sabe descrever o produto em 2-3 parágrafos
  • Não há multi-tenancy, permissões complexas ou integrações externas
  • Tolerância alta para retrabalho futuro

🏗️ Use o BMAD (KERNO-ROADMAP) se:

  • 4+ features core ou escopo que vai crescer
  • Produto com usuários reais pagantes
  • Multi-tenancy, papéis, permissões granulares
  • Integração com sistemas externos (Stripe, APIs)
  • Você quer que o Lovable respeite uma arquitetura definida
  • Baixa tolerância para retrabalho
💡 Modo híbrido válido: comece com Lovable-First, e se o produto validar, faça o BMAD retroativamente — Mary faz o product-brief a partir do que já existe, Winston documenta a arquitetura atual. Não é obrigatório refazer do zero.

⚡ Sua Jornada em 5 Etapas

01 PRD Mínimo — Escreva antes de abrir o Lovable 15 min 02 Ambiente Seguro — GitHub + SAST antes do Lovable 15 min 03 Desenvolvimento — Ciclo Lovable com prompts prontos contínuo 04 Revisão de Segurança — Antes de qualquer usuário real 30 min 05 Deploy — Vercel + monitoramento 20 min
Diferença em relação ao guia completo: sem Etapa 00 (BMAD), sem revisão Cursor. O Lovable faz o planejamento interno no Plan Mode, baseado no seu PRD manual. Você assume a responsabilidade de escopo.
ETAPA 01 PRD Mínimo ⏱️ 15 min · antes de abrir o Lovable

Antes de digitar o primeiro prompt no Lovable, escreva o PRD mínimo. Sem isso, o Lovable inventa a arquitetura — e você vai gastar horas corrigindo decisões ruins de banco de dados. 15 minutos de escrita economizam 5 horas de retrabalho.

Passo 1.1

📝 Crie o arquivo prd.md

Crie uma pasta para o seu projeto e dentro dela uma subpasta docs/. Crie o arquivo docs/prd.md com o template abaixo. Preencha os campos marcados com [...] — os demais são instruções para o Lovable e ficam exatamente assim.

docs/prd.md — template
# PRD — [Nome do Produto]

## Visão geral
[Descreva em 2-4 frases o que é o produto e para quem é.
Ex: "App web para personal trainers gerenciarem fichas de treino de alunos.
O personal cria fichas, associa exercícios e o aluno consulta pelo celular."]

## Usuários
- **Usuário principal:** [quem vai usar — papel e contexto]
- **Usuário secundário (se houver):** [outro papel, se existir]

## Funcionalidades do MVP (em ordem de prioridade)
1. [Feature mais importante — seja específico. Ex: "Cadastro e login de trainers e alunos"]
2. [Segunda feature]
3. [Terceira feature]
[Adicione mais se necessário. Máximo 5-6 para um MVP real.]

## Funcionalidades FORA do escopo (não implementar agora)
- [Feature que pode parecer óbvia mas não é para o MVP]
- [Outra funcionalidade que vem depois]
[Esta seção é tão importante quanto as de cima. Diga ao Lovable o que NÃO fazer.]

## Dados que o sistema precisa guardar
[Liste os "objetos" principais. Ex: "trainers (nome, email, especialidade), alunos
(nome, email, trainer responsável), fichas de treino (título, data, lista de exercícios),
exercícios (nome, séries, repetições, carga, observações)"]

## Requisitos não-negociáveis
- Autenticação obrigatória — nenhuma tela acessível sem login
- Cada trainer só vê seus próprios alunos e fichas (RLS no Supabase)
- Funcionar bem no celular (mobile-first)
- [Adicione outros se houver — ex: "exportar PDF", "funcionar offline"]

## Stack
- Frontend: React + TypeScript + Tailwind + shadcn/ui
- Backend: Supabase (Auth + PostgreSQL + RLS)
- Deploy: Vercel
💡 Por que o seção "fora do escopo" é importante?

O Lovable é empolgado. Se você não diz o que não quer, ele vai adicionar funcionalidades que pareceram óbvias para ele — e você vai ter código que não pediu, bugs que não entende, e um app maior do que precisa. Dizer "NÃO" é tão importante quanto dizer "SIM".

Passo 1.2

🗺️ Pense nas milestones antes de começar

Você não precisa escrever um prd-milestones.md formal (o Lovable vai criar isso). Mas antes de começar, tenha clareza mental sobre a ordem:

  • Milestone 1: Auth + layout base (sempre primeiro — adicionar auth depois quebra tudo)
  • Milestone 2: Schema do banco + RLS (o Supabase não reverte bem — planejamento aqui é crítico)
  • Milestone 3: Feature principal (o core do produto)
  • Milestone 4+: Features secundárias, polish, integrações
⚠️ Armadilha comum do modo Lovable-First: pedir features antes de ter auth e banco configurados. O Lovable vai aceitar — e depois você vai passar horas refazendo porque a estrutura estava errada. Auth SEMPRE primeiro.
✅ Gate — Saída da Etapa 01
  • Arquivo docs/prd.md criado e preenchido
  • Funcionalidades fora do escopo listadas
  • Você sabe descrever o produto em 2 frases para qualquer pessoa
ETAPA 02 Ambiente Seguro ⏱️ 15 min · antes do primeiro prompt

Esta etapa é idêntica ao guia completo — não há atalho aqui. Pipeline SAST e Seção A no Lovable são os dois pilares de segurança que precisam estar no lugar desde o início. Segurança retroativa é mais cara que segurança desde o dia 1.

Passo 2.1

🔗 Criar repositório GitHub privado

  1. Acesse github.com → New repository
  2. Nome: o nome do seu projeto (minúsculas, sem espaços)
  3. Visibilidade: Private
  4. Crie o repositório
Passo 2.2

🔒 Adicionar pipeline SAST + Dependabot

No GitHub, adicione os dois arquivos de segurança do Kerno:

  1. Vá em Add file → Create new file
  2. Nome: .github/workflows/security-audit.yml
  3. Cole o conteúdo do arquivo security-audit-v3.yml do Kerno
  4. Repita para .github/dependabot.yml
security-audit-v3.yml dependabot.yml
⚠️ Não pule este passo. O pipeline roda em todo push automaticamente. Sem ele, você não tem como saber se o Lovable está criando vulnerabilidades — e ele cria, às vezes.
Passo 2.3

⚙️ Configurar o Lovable — Seção A

No Lovable, vá em Settings → Knowledge e cole o conteúdo de KERNO-KNOWLEDGE.md. Esta seção define as regras de segurança, documentação e ordem de implementação que o Lovable vai seguir em todo o projeto.

LOVABLE-SYSTEM-PROMPT-V3.md — Seção A
💡 O que a Seção A faz:

São as regras permanentes do seu projeto. Elas ficam em todo prompt sem você precisar repetir. Elas dizem ao Lovable: sempre use RLS, nunca use service_role no frontend, atualize a documentação em todo prompt, siga a ordem auth → dados → features. É o "contrato de trabalho" da IA.

Passo 2.4

📤 Conectar Lovable ao GitHub + Upload do PRD

  1. No Lovable, conecte ao repositório GitHub que você criou
  2. Faça upload do docs/prd.md no Knowledge do Lovable
  3. Faça upload do KERNO-DOCS.md e SECURITY.md do Kerno
KERNO-DOCS.md SECURITY.md
✅ Gate — Saída da Etapa 02
  • Repositório GitHub privado criado
  • security-audit-v3.yml e dependabot.yml no GitHub (aba Actions visível)
  • KERNO-KNOWLEDGE.md colado no Custom Knowledge do Lovable
  • Lovable conectado ao GitHub
  • prd.md, KERNO-DOCS.md e SECURITY.md no Knowledge do Lovable
ETAPA 03 Desenvolvimento ⏱️ Contínuo · ciclo de prompts

O ciclo de desenvolvimento Lovable-First usa o mesmo conjunto de prompts do guia completo (Seções B-F), mas com uma diferença crítica: na Seção B, o Lovable vai criar os milestones a partir do seu PRD manual — não importar milestones do BMAD. Você vai ver e aprovar o plano antes de qualquer código.

Passo 3.1 — Prompt de Início de Projeto

🚀 Seção B — Iniciar projeto

Vá ao chat do Lovable e cole o prompt abaixo. Use Plan Mode se disponível — o Lovable vai planejar sem escrever código:

Leia o prd.md e o KERNO-DOCS.md que enviei no Knowledge.

Use o Plan Mode. Antes de implementar qualquer código, faça:

1. Analise o prd.md e entenda o produto, usuários e escopo

2. Crie /docs/prd-milestones.md com as milestones seguindo
   OBRIGATORIAMENTE a Árvore de Dependências do Custom Knowledge:
   - Milestone 1: Auth + layout base (login, registro, proteção de rotas, sidebar/header)
   - Milestone 2: Banco de dados (schema completo, RLS em todas as tabelas, policies)
   - Milestone 3: Feature core (a funcionalidade principal do prd.md)
   - Milestone 4+: Features secundárias em ordem de prioridade do prd.md
   
   Cada milestone deve ter:
   - Features incluídas (do prd.md)
   - Critérios de aceite objetivos
   - Dependências (o que precisa estar pronto antes)

3. Crie os documentos operacionais que ainda não existem:
   /docs/ui-components.md (estrutura inicial)
   /docs/flows.md (estrutura inicial)
   /docs/changelog.md (v0.1.0 — início do projeto)
   /docs/handover.md (estrutura inicial)
   /docs/security-rules.md (baseado no SECURITY.md)
   /docs/documentation-rules.md
   /docs/session-close.md (estrutura)
   /docs/delivery-report.md (estrutura)
   /docs/security-report.md (estrutura)

4. Apresente o plano da Milestone 1 para minha aprovação ANTES de implementar.

Não implemente nada ainda. Só planeje, crie os docs, e me mostre o plano.
💡 O que vai acontecer:

O Lovable vai ler o seu PRD e criar um plano organizado em milestones. Ele vai mostrar quais features entram em cada fase. Leia com atenção — se ele incluiu algo que você não quer ou colocou na ordem errada, corrija antes de aprovar. Isso é muito mais fácil agora do que depois.

Passo 3.2 — Prompt de Aprovação

✅ Seção C — Aprovar e começar Milestone 1

Quando o plano estiver bom, cole:

O plano está aprovado. Implemente a Milestone 1 agora.

Lembrete:
- Auth SEMPRE primeiro (login, registro, proteção de rotas)
- Depois layout base (header, sidebar, navegação)
- Mobile-first
- RLS em toda tabela nova com policies para auth.uid() = user_id
- Valide formulários com Zod
- Atualize /docs a cada passo
- Ao final, avise para fixar a versão
⚠️ Se o plano NÃO estiver bom: diga o que quer mudar antes de aprovar. Exemplo: "Remova pagamentos da Milestone 3, isso é pós-MVP" ou "Adicione a exportação de PDF na Milestone 4". Ajuste agora — depois é mais caro.
Passo 3.3 — Ciclo de sessões

🔄 Seção D — Retomar sessão

Ao retomar o trabalho depois de fechar o navegador, cole este prompt antes de qualquer outro:

Retomando sessão. Contexto:

- Última sessão: [data]
- Milestone atual: [número] — [X]% concluída
- Pendências: [copie o conteúdo de /docs/session-close.md]

Continue a implementação. Siga a ordem de dependências
do Custom Knowledge. Atualize /docs conforme o KERNO-DOCS.md.
💡 Por que copiar o session-close.md?

O Lovable não tem memória entre sessões. O session-close.md é o "caderno de anotações" — ele guarda onde você parou, o que está pendente, decisões tomadas. Sem ele, o Lovable começa do zero e pode contradizer o que já foi feito.

Passo 3.4 — Fechar sessão

💾 Seção E — Fechar sessão

Ao terminar de trabalhar por hoje, sempre feche a sessão com:

Fechar sessão.

Atualize /docs/session-close.md com:
- O que foi implementado hoje
- O que está pendente para a próxima sessão
- Decisões técnicas tomadas
- Problemas encontrados e como foram resolvidos

Atualize também /docs/changelog.md com as mudanças desta sessão.
Passo 3.5 — Fechar milestone

🏁 Seção F — Fechar milestone

Ao concluir uma milestone completa:

Fechar Milestone [N].

1. Atualize /docs/prd-milestones.md marcando a Milestone [N] como concluída
2. Gere /docs/delivery-report.md com:
   - Escopo planejado vs. entregue
   - Features implementadas com status
   - Issues conhecidos (se houver)
   - Recomendação: pronto para avançar para Milestone [N+1]?
3. Atualize /docs/changelog.md

Avise quando terminar — vou fixar a versão antes de iniciar a próxima milestone.
⚠️ Fixar versão antes de iniciar a próxima milestone é obrigatório. No histórico do Lovable, clique em "pin" na versão atual. Se a próxima milestone quebrar algo, você consegue voltar.
Passo 3.6 — QA durante desenvolvimento

🧪 Teste enquanto constrói

A cada milestone, teste manualmente no preview do Lovable:

  • Abra uma aba anônima e tente acessar páginas protegidas sem login — deve redirecionar
  • Crie dois usuários diferentes e verifique que cada um só vê seus próprios dados
  • Teste em mobile (redimensione o navegador para <375px)
  • Abra DevTools → Console — não deve ter erros visíveis

Para QA mais profundo, use este prompt:

Faça uma revisão de segurança da Milestone [N] que acabamos de implementar.

Verifique:
1. RLS ativo em todas as tabelas novas? Liste cada tabela e status
2. Algum service_role usado no frontend? Onde?
3. Formulários com validação Zod? Quais ainda faltam?
4. Rotas protegidas? Listar quais precisam de auth
5. Dados sensíveis em localStorage? Listar se houver

Para cada problema encontrado, proponha e implemente a correção.
Atualize /docs/security-report.md ao final.
✅ Gate — Por milestone (antes de avançar)
  • Critérios de aceite da milestone conferidos contra prd-milestones.md
  • delivery-report.md gerado: escopo planejado = entregue
  • Versão fixada (pin) no Lovable
  • Pipeline SAST sem findings críticos (aba Actions verde no GitHub)
  • Teste de acesso não-autorizado passou (aba anônima não entra em páginas protegidas)
ETAPA 04 Revisão de Segurança ⏱️ 30 min · antes de qualquer usuário real

Antes de dar acesso a qualquer pessoa real — mesmo em beta privado — passe pelo checklist de segurança. No modo Lovable-First, você não teve Murat (TEA) nem revisão no Cursor, então esta etapa compensa parcialmente essa ausência.

Passo 4.1

🔍 SAST — Verificar resultado do pipeline

Acesse GitHub → seu repositório → Actions. Veja o resultado do último workflow:

  • ✅ Verde: sem findings críticos — pode avançar
  • ❌ Vermelho: abra o log e veja o que falhou

Se houver findings críticos, cole-os no Lovable:

O pipeline SAST encontrou os seguintes problemas:

[cole aqui o output do GitHub Actions]

Analise cada finding e implemente as correções necessárias.
Priorize por severidade: Critical > High > Medium > Low.
Atualize /docs/security-report.md ao final.
Passo 4.2

🗄️ Supabase — Verificação manual de RLS

No painel do Supabase, vá em Table Editor → cada tabela → Policies. Verifique para cada tabela que contém dados de usuário:

VerificaçãoOndeStatus
RLS ativoTable Editor → cada tabela⬜
Policy SELECT: auth.uid() = user_idPolicies de cada tabela⬜
Policy INSERT com user_id corretoPolicies de cada tabela⬜
Email confirmation ativoAuth → Settings⬜
Nenhum bucket de Storage público com dados sensíveisStorage → buckets⬜
⚠️ RLS desabilitado é a vulnerabilidade mais comum em apps Lovable. Qualquer usuário autenticado consegue ver os dados de todos os outros. Verifique manualmente — o scanner não acessa o banco diretamente.
Passo 4.3

🌐 Testes dinâmicos manuais

Com a app no preview, execute estes testes — são os que o Lovable não consegue fazer por você:

Checklist de segurança — executar antes de qualquer usuário real

⬜ Aba anônima: acessar URL de página protegida diretamente → redireciona para login?
⬜ Criar usuário A e usuário B → usuário A não vê dados de B?
⬜ Tentar mudar ID na URL para ver dados de outro usuário → bloqueado?
⬜ Digitar <script>alert(1)</script> em campos de texto → não executa?
⬜ DevTools → Console → não mostra dados sensíveis ou tokens de API?
⬜ DevTools → Application → LocalStorage → não tem senhas, tokens, dados de usuário?
⬜ Tentar enviar formulário vazio em campo obrigatório → validação aparece?
Passo 4.4 — Opcional mas recomendado

🤖 Revisão TEA com Murat (se tiver Claude Code)

Se você tiver o BMAD instalado, pode usar o Murat agora — retroativamente — para criar um qa-plan baseado no código que existe:

@murat Analise o projeto que já foi implementado.

Leia /docs/prd.md para entender o escopo.
Leia /docs/prd-milestones.md para entender o que foi construído.

Crie /docs/qa-plan.md retroativo com:
1. Classifique as features existentes em P0 a P3 por risco
2. Para cada P0: liste casos de teste manuais obrigatórios
3. Checklist de segurança específico para este projeto
4. Critério objetivo de "pronto para deploy"
🔮 Quando considerar migrar para o fluxo completo: Se o produto validou e você está prestes a ter usuários reais pagantes, este é o momento certo para fazer o BMAD retroativamente. Mary pode fazer o product-brief a partir do que já existe. Winston pode documentar a arquitetura real. O Murat pode criar o qa-plan definitivo. O custo de fazer agora é muito menor do que depois de ter 50 usuários.
✅ Gate — Saída da Etapa 04 (antes de qualquer usuário real)
  • SAST verde no GitHub Actions
  • RLS ativo em 100% das tabelas com dados de usuário
  • Sem service_role no frontend (Scanner Kerno passou)
  • Todos os 7 itens do checklist manual acima verificados
  • Acesso não-autorizado bloqueado em teste manual
ETAPA 05 Deploy ⏱️ 20 min · quando a revisão de segurança passar

Deploy idêntico ao guia completo. Vercel para hospedagem, headers de segurança obrigatórios, Supabase em configuração de produção, e monitoramento básico.

Passo 5.1

▲ Vercel — Headers de segurança

Antes de fazer o deploy, peça ao Lovable para criar o vercel.json:

Crie o arquivo vercel.json na raiz do projeto com os headers de segurança
do Kerno Framework (X-Frame-Options, X-Content-Type-Options, HSTS, CSP).
Use os valores padrão do Kerno para um projeto React + Supabase.

Depois do deploy, verifique em securityheaders.com — deve mostrar nota A ou B.

Passo 5.2

🗄️ Supabase — Configuração de produção

ConfiguraçãoOnde
Email confirmation: ONAuth → Settings
Site URL: sua URL de produçãoAuth → URL Configuration
Redirect URLs: adicionar domínio de produçãoAuth → URL Configuration
Backups: ativar Point-in-Time RecoveryDatabase → Backups
Passo 5.3

📡 Monitoramento mínimo

Dois serviços gratuitos que você precisa ter antes de qualquer usuário real:

  • Sentry (sentry.io — plano gratuito) — detecta erros em produção automaticamente
  • UptimeRobot (uptimerobot.com — plano gratuito) — monitora se a app está no ar

Para adicionar o Sentry ao projeto:

Adicione o Sentry ao projeto para monitoramento de erros em produção.
DSN do Sentry: [cole aqui o DSN que o Sentry gerou para você]

Configure para capturar erros em produção mas não em desenvolvimento local.
Não capture dados pessoais dos usuários nas mensagens de erro.
✅ Gate — Deploy completo
  • App acessível na URL de produção sem erros
  • securityheaders.com mostra nota A ou B
  • Supabase configurado para produção (email confirmation ON, Site URL correto)
  • Sentry recebendo eventos (faça um erro de teste)
  • UptimeRobot monitorando (criou monitor HTTP para a URL)

⚠️ Limites deste modo — quando migrar para o fluxo completo

O modo Lovable-First funciona bem para MVPs rápidos e validação de ideia. À medida que o produto cresce, estes sinais indicam que chegou a hora de adotar o BMAD:

SinalO que acontece sem BMADSolução
O Lovable está "esquecendo" decisões antigasContradições no código, features reimplementadas de formas diferentesWinston retroativo: documenta a arquitetura real
Você não sabe mais o que está em qual milestoneEscopo descontrolado, features incompletas por todo ladoJohn retroativo: reestrutura milestones a partir do código existente
Schema do banco com tabelas desnecessárias ou estrutura estranhaQueries complexas, RLS difícil de manterWinston retroativo + migração de schema
Primeiro usuário pagante está chegandoSem QA plan formal, sem critério de "pronto para produção"Murat retroativo: cria qa-plan a partir do que existe
Time de 2+ pessoas vai mexer no projetoConflitos de decisão, sem onboarding novo membroFluxo BMAD completo: a documentação é o onboarding
🔮 O BMAD retroativo existe. Você não precisa reconstruir do zero. Leia a seção "E os agentes BMAD?" no KERNO-RETROFIT.html para entender como fazer a transição.
