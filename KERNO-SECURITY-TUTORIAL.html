01 Conceitos Fundamentais

Antes de configurar qualquer ferramenta, vamos entender os conceitos. Isso é essencial para saber por que cada ferramenta existe e quando usar cada uma.

🔍 SAST — Static Application Security Testing

Analogia: Um revisor que lê o projeto arquitetônico de uma casa e aponta falhas ANTES de construir. Ele não precisa ver a casa pronta — analisa os desenhos.

O que faz: Analisa o código-fonte sem executar a aplicação. Procura padrões perigosos como XSS, injection, secrets, etc.

Quando roda: A cada push no GitHub (automático).

Ferramenta: Semgrep + Scanner customizado Kerno.

🌐 DAST — Dynamic Application Security Testing

Analogia: Um ladrão contratado que tenta invadir a casa DEPOIS de construída. Ele testa as portas, janelas, e tenta entrar de verdade.

O que faz: Testa a aplicação rodando — tenta acessar URLs sem login, manipular dados, injetar código, etc.

Quando roda: Manualmente antes do deploy e periodicamente.

Ferramenta: Checklist manual + testes com DevTools do navegador.

📊 Monitoramento de Erros

Analogia: Uma câmera de segurança dentro da casa que grava quando algo quebra. Se uma janela racha, você recebe um alerta no celular imediatamente.

O que faz: Captura erros JavaScript em tempo real, mostrando exatamente o que aconteceu, onde, e com qual usuário.

Quando roda: 24/7 enquanto a aplicação está no ar.

Ferramenta: Sentry (plano gratuito).

📝 Logs de Auditoria

Analogia: Um livro de registro na portaria do prédio. Anota quem entrou, quando, o que fez, e quando saiu. Se algo desaparecer, você sabe quem esteve lá.

O que faz: Registra quem acessou o quê, quem alterou dados, quem tentou acessar algo não permitido.

Quando roda: 24/7 — cada ação no banco é registrada.

Ferramenta: Tabela de audit_log no Supabase + trigger automático.

🛡️ WAF — Web Application Firewall

Analogia: Um segurança na porta do prédio que verifica a identidade de cada visitante. Se alguém parece suspeito (tentando invadir), é barrado antes de entrar.

O que faz: Filtra requisições maliciosas (SQL injection, XSS, bots, DDoS) ANTES de chegarem à sua aplicação.

Quando roda: 24/7 — fica entre o visitante e seu servidor.

Ferramenta: Cloudflare (plano gratuito).

⏱️ Monitoramento de Uptime

Analogia: Alguém que liga para sua casa a cada 5 minutos para ver se o telefone ainda funciona. Se não atender, você recebe um alerta.

O que faz: Verifica periodicamente se sua aplicação está no ar e respondendo.

Quando roda: A cada 1-5 minutos, 24/7.

Ferramenta: UptimeRobot ou Better Stack (gratuitos).

02 SAST — Implementação Detalhada

Você já tem o arquivo security-audit-v3.yml do Kerno. Aqui vai o passo a passo detalhado para implementar.

📁 2.1 Criar o pipeline no GitHub
  1. Abra seu repositório no GitHub
  2. Clique em Add file → Create new file
  3. No campo de nome, digite: .github/workflows/security-audit.yml
  4. Cole o conteúdo completo do arquivo security-audit-v3.yml
  5. Clique em Commit changes
💡 O pipeline roda 3 ferramentas em paralelo: Semgrep (análise estática), npm audit (dependências) e o Scanner customizado do Kerno (Supabase, secrets, auth).
📦 2.2 Configurar Dependabot
  1. No mesmo repositório, Add file → Create new file
  2. Nome: .github/dependabot.yml
  3. Cole:
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule:
      interval: "weekly"
      day: "monday"
    open-pull-requests-limit: 5
    labels:
      - "security"
      - "dependencies"

03 DAST — Testes com a Aplicação Rodando

DAST testa a aplicação de verdade, como um atacante faria. Para seu cenário (sem equipe de pentest), vamos usar testes manuais estruturados que você pode fazer pelo navegador.

🌐 3.1 Teste de Acesso Não Autorizado

Objetivo: Verificar se alguém sem login consegue acessar dados.

  1. Abra o navegador em modo anônimo (Ctrl+Shift+N)
  2. Tente acessar URLs protegidas diretamente (ex: /dashboard, /admin, /profile)
  3. Esperado: Redirecionar para login em TODAS
  4. Se qualquer página mostrar conteúdo sem login → FALHA CRÍTICA
🔄 3.2 Teste de IDOR (Acesso a dados de outro usuário)

Objetivo: Verificar se um usuário consegue ver dados de outro.

  1. Faça login com Usuário A
  2. Crie um registro (documento, perfil, etc.) — anote o ID na URL
  3. Faça logout e login com Usuário B
  4. Tente acessar o registro do Usuário A (mudando o ID na URL)
  5. Esperado: Erro 403 ou redirecionamento. NUNCA mostrar os dados.
🚨 Se o Usuário B conseguir ver dados do Usuário A, suas RLS policies estão incorretas. Corrija imediatamente no Supabase.
💉 3.3 Teste de XSS (Injeção de Script)

Objetivo: Verificar se a aplicação executa código malicioso.

  1. Em CADA campo de texto do app (nome, descrição, comentário), digite: <script>alert('XSS')</script>
  2. Salve e veja se aparece um popup de alerta
  3. Teste também: <img src=x onerror=alert('XSS')>
  4. Esperado: O texto deve aparecer como texto puro, NUNCA executar o script
🔎 3.4 Verificar Console e Network

Objetivo: Verificar se dados sensíveis estão vazando.

  1. Abra DevTools (F12) → aba Console
  2. Navegue pela aplicação normalmente
  3. Procure por: senhas, tokens, CPFs, dados de cartão no console
  4. Vá para a aba Network
  5. Verifique as requisições — elas estão enviando dados desnecessários?
  6. Verifique se TODAS as URLs são HTTPS (sem HTTP)
  7. Esperado: Console limpo, sem dados sensíveis. Só HTTPS.
🔐 3.5 Teste de Brute Force

Objetivo: Verificar se o login aceita tentativas infinitas.

  1. Tente fazer login 10x com senha errada
  2. Esperado: Após 5-10 tentativas, o Supabase deve bloquear temporariamente
  3. Verifique em: Supabase → Auth → Rate Limits
📱 3.6 Teste de Responsividade

Abra DevTools → toggle device toolbar (Ctrl+Shift+M). Teste em: iPhone SE, iPhone 14, iPad, desktop.

⚠️ Faça esses testes ANTES de cada deploy para produção e sempre que adicionar uma nova feature que envolva dados de usuário.
04 Monitoramento de Erros — Sentry

📊 Sentry

Plataforma de monitoramento que captura erros em tempo real, mostrando o stack trace, navegador, usuário e ações que levaram ao erro.
Plano gratuito disponível 5K erros/mês no free React/Vite suportado
🔧 4.1 Criar conta no Sentry
  1. Acesse sentry.io → Sign Up (use o GitHub para facilitar)
  2. Crie uma organização (pode ser o nome da sua agência)
  3. Crie um projeto: escolha React como plataforma
  4. Copie o DSN (é uma URL tipo https://[email protected]/yyyy)
📦 4.2 Instalar no projeto via Lovable
Instale o Sentry para monitoramento de erros em produção.

1. Adicione a dependência @sentry/react
2. Crie um arquivo src/lib/sentry.ts que inicializa o Sentry:
   - Use import.meta.env.VITE_SENTRY_DSN para o DSN
   - Habilite BrowserTracing e Replay
   - Sample rate: 1.0 para errors, 0.1 para transactions
   - Só ative em produção (import.meta.env.PROD)
3. Importe e inicialize no main.tsx ANTES do React render
4. Envolva o App com Sentry.ErrorBoundary para capturar erros de componentes
5. NÃO envie dados sensíveis ao Sentry (filtre PII com beforeSend)

Não altere nenhuma funcionalidade existente.
🔑 4.3 Configurar variável de ambiente

Na Vercel:

  1. Settings → Environment Variables
  2. Adicione: VITE_SENTRY_DSN = [o DSN que copiou]
  3. Ambiente: Production only
  4. Redeploy para ativar
🔔 4.4 Configurar alertas

No painel do Sentry:

  1. Alerts → Create Alert Rule
  2. Tipo: "When a new issue is created"
  3. Ação: Send email to me
  4. Crie outro alerta para: "When event frequency exceeds 10 in 1 hour" (pode indicar ataque)

05 Logs de Auditoria — Quem Fez o Quê

Logs de auditoria registram CADA ação no seu banco de dados: quem criou, editou, ou deletou dados, e quando. Se alguém acessar indevidamente, você terá a prova.

🗄️ 5.1 Criar tabela de audit_log no Supabase

No painel do Supabase → SQL Editor, execute:

SQL — Criar tabela de auditoria
-- Tabela que registra cada ação no banco
CREATE TABLE IF NOT EXISTS audit_log (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  timestamp TIMESTAMPTZ DEFAULT now() NOT NULL,
  user_id UUID REFERENCES auth.users(id),
  action TEXT NOT NULL,        -- 'INSERT', 'UPDATE', 'DELETE'
  table_name TEXT NOT NULL,    -- qual tabela foi afetada
  record_id TEXT,              -- ID do registro afetado
  old_data JSONB,              -- dados antes da mudança
  new_data JSONB,              -- dados depois da mudança
  ip_address TEXT              -- IP do usuário (quando disponível)
);

-- RLS: só admins podem VER os logs, ninguém pode editar/deletar
ALTER TABLE audit_log ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Admins can view audit logs"
  ON audit_log FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM user_roles
      WHERE user_roles.user_id = auth.uid()
      AND user_roles.role = 'admin'
    )
  );

-- Ninguém pode deletar ou editar logs (nem admin)
-- A ausência de policies para DELETE/UPDATE impede essas operações

-- Index para buscas rápidas
CREATE INDEX idx_audit_log_timestamp ON audit_log(timestamp DESC);
CREATE INDEX idx_audit_log_user ON audit_log(user_id);
CREATE INDEX idx_audit_log_table ON audit_log(table_name);
⚡ 5.2 Criar trigger automático

Este trigger registra automaticamente toda alteração nas tabelas que você escolher:

SQL — Função de auditoria genérica
-- Função que registra mudanças automaticamente
CREATE OR REPLACE FUNCTION log_audit()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO audit_log (
    user_id,
    action,
    table_name,
    record_id,
    old_data,
    new_data
  ) VALUES (
    auth.uid(),
    TG_OP,
    TG_TABLE_NAME,
    COALESCE(NEW.id::text, OLD.id::text),
    CASE WHEN TG_OP IN ('UPDATE','DELETE') THEN to_jsonb(OLD) ELSE NULL END,
    CASE WHEN TG_OP IN ('INSERT','UPDATE') THEN to_jsonb(NEW) ELSE NULL END
  );

  IF TG_OP = 'DELETE' THEN
    RETURN OLD;
  ELSE
    RETURN NEW;
  END IF;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
🔗 5.3 Conectar às tabelas sensíveis

Para CADA tabela que contenha dados sensíveis, conecte o trigger:

SQL — Ativar auditoria nas tabelas
-- Substitua 'sua_tabela' pelo nome real de cada tabela
-- Repita para CADA tabela com dados sensíveis

CREATE TRIGGER audit_sua_tabela
  AFTER INSERT OR UPDATE OR DELETE ON sua_tabela
  FOR EACH ROW EXECUTE FUNCTION log_audit();

👀 5.4 Como consultar os logs

No SQL Editor do Supabase:

SQL — Consultas úteis
-- Últimas 50 ações
SELECT * FROM audit_log ORDER BY timestamp DESC LIMIT 50;

-- Ações de um usuário específico
SELECT * FROM audit_log
WHERE user_id = 'UUID_DO_USUARIO'
ORDER BY timestamp DESC;

-- Todas as deleções (suspeito)
SELECT * FROM audit_log
WHERE action = 'DELETE'
ORDER BY timestamp DESC;

-- Ações nas últimas 24 horas
SELECT * FROM audit_log
WHERE timestamp > now() - interval '24 hours'
ORDER BY timestamp DESC;

-- Ações em uma tabela específica
SELECT * FROM audit_log
WHERE table_name = 'transactions'
ORDER BY timestamp DESC;
⏰ 5.5 Limpeza automática de logs (Cron Job) — Tutorial completo

Os audit logs vão crescer indefinidamente. Sem limpeza, o banco fica lento e caro. Vamos configurar uma limpeza automática semanal.

🤔 O que é um Cron Job?

Analogia: É como programar o despertador para tocar toda segunda-feira às 3h da manhã e fazer uma faxina automática na casa. Você configura uma vez, e ele repete sozinho para sempre.

"Cron" vem do Unix — é um agendador de tarefas. A expressão 0 3 * * 1 significa "às 3:00, toda segunda-feira". Cada posição representa: minuto, hora, dia do mês, mês, dia da semana.

  • 0 3 * * 1 → Toda segunda às 3h
  • 0 0 * * * → Todo dia à meia-noite
  • 0 12 1 * * → Todo dia 1 ao meio-dia
🔧 5.5.1 Opção A — pg_cron no Supabase (recomendado)

pg_cron é uma extensão do PostgreSQL que roda tarefas agendadas direto no banco. No Supabase, você ativa pelo painel.

Passo a passo:

  1. Abra Supabase → Database → Extensions
  2. Procure pg_cron e ative
  3. Vá para SQL Editor e execute os comandos abaixo, UM de cada vez:
SQL — Passo 1: Criar a função de limpeza
-- Função que limpa logs antigos
-- INSERT/UPDATE: mantém 90 dias
-- DELETE: mantém 365 dias (mais sensível)
CREATE OR REPLACE FUNCTION cleanup_audit_logs()
RETURNS void AS $$
DECLARE
  removed_short INTEGER;
  removed_long INTEGER;
BEGIN
  -- Contar antes de deletar (para registro)
  SELECT count(*) INTO removed_short
  FROM audit_log
  WHERE action IN ('INSERT', 'UPDATE')
  AND timestamp < now() - interval '90 days';

  SELECT count(*) INTO removed_long
  FROM audit_log
  WHERE action = 'DELETE'
  AND timestamp < now() - interval '365 days';

  -- Deletar logs antigos
  DELETE FROM audit_log
  WHERE action IN ('INSERT', 'UPDATE')
  AND timestamp < now() - interval '90 days';

  DELETE FROM audit_log
  WHERE action = 'DELETE'
  AND timestamp < now() - interval '365 days';

  -- Registrar a limpeza no próprio audit_log
  INSERT INTO audit_log (user_id, action, table_name, record_id, new_data)
  VALUES (
    NULL,
    'SYSTEM',
    'audit_log',
    'cleanup',
    jsonb_build_object(
      'insert_update_removed', removed_short,
      'delete_removed', removed_long,
      'executed_at', now()
    )
  );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
SQL — Passo 2: Agendar o cron job
-- Agendar limpeza toda segunda-feira às 3h UTC
SELECT cron.schedule(
  'cleanup-audit-logs',    -- nome do job
  '0 3 * * 1',             -- toda segunda às 3h UTC
  'SELECT cleanup_audit_logs()'
);
SQL — Verificar que o cron foi criado
-- Listar todos os cron jobs agendados
SELECT * FROM cron.job;
SQL — Testar manualmente (opcional)
-- Para testar se funciona, rode manualmente:
SELECT cleanup_audit_logs();

-- Depois verifique o registro da limpeza:
SELECT * FROM audit_log
WHERE action = 'SYSTEM'
ORDER BY timestamp DESC
LIMIT 5;
SQL — Comandos úteis para gerenciar cron jobs
-- Ver todos os jobs
SELECT * FROM cron.job;

-- Ver histórico de execuções
SELECT * FROM cron.job_run_details ORDER BY start_time DESC LIMIT 10;

-- Pausar um job (sem deletar)
SELECT cron.unschedule('cleanup-audit-logs');

-- Re-agendar (mudar frequência)
-- Primeiro desagende, depois agende de novo:
SELECT cron.unschedule('cleanup-audit-logs');
SELECT cron.schedule('cleanup-audit-logs', '0 3 * * 1', 'SELECT cleanup_audit_logs()');
🌐 6.1.0 Opção B — Cron externo (se pg_cron não estiver disponível)

Se seu plano do Supabase não tem pg_cron, use um serviço externo gratuito:

  1. No Supabase, crie uma Edge Function (menu Functions) com o mesmo código de limpeza
  2. Acesse cron-job.org → crie conta gratuita
  3. Crie um novo cron job:
    • URL: a URL da sua Edge Function
    • Frequência: Semanal, segunda-feira, 03:00
    • Método: POST
    • Header: Authorization: Bearer [sua_anon_key]
⚠️ A Opção A (pg_cron) é mais segura porque roda dentro do banco. A Opção B expõe a Edge Function externamente — proteja com verificação de API key.
✅ 6.1.0 Verificar que está funcionando

Após uma semana, verifique:

SQL — Verificar execuções do cron
-- Ver se o cron rodou e quantos logs removeu
SELECT * FROM audit_log
WHERE action = 'SYSTEM'
AND table_name = 'audit_log'
ORDER BY timestamp DESC;

-- Se pg_cron, ver histórico de execuções
SELECT * FROM cron.job_run_details
WHERE jobid = (SELECT jobid FROM cron.job WHERE jobname = 'cleanup-audit-logs')
ORDER BY start_time DESC
LIMIT 5;

📊 6.2 Dashboard de segurança no Supabase

Crie uma view no Supabase que mostra anomalias:

SQL — View de anomalias
-- Criar view para análise rápida
CREATE OR REPLACE VIEW security_dashboard AS

-- Ações nas últimas 24h por tipo
SELECT
  'ações_24h' as metric,
  action as detail,
  count(*) as value
FROM audit_log
WHERE timestamp > now() - interval '24 hours'
GROUP BY action

UNION ALL

-- Usuários mais ativos nas últimas 24h
SELECT
  'usuarios_ativos_24h' as metric,
  COALESCE(user_id::text, 'anônimo') as detail,
  count(*) as value
FROM audit_log
WHERE timestamp > now() - interval '24 hours'
GROUP BY user_id
ORDER BY value DESC
LIMIT 10;

-- Consultar:
-- SELECT * FROM security_dashboard;

Crie uma Supabase Edge Function chamada "security-alert"
que é chamada por um database webhook quando:

- Um usuário faz mais de 100 ações em 1 hora

A função deve enviar um email de alerta para meu email

08 Proteção contra Ataques — Cloudflare

☁️ Cloudflare

CDN + WAF + DDoS protection. Fica entre o usuário e seu servidor, filtrando tráfego malicioso.
Plano gratuito: CDN + DDoS + SSL + WAF básico ~15 min para configurar
🔧 8.1 Configurar Cloudflare

Pré-requisito: Ter um domínio customizado (não funciona com .vercel.app).

  1. Crie conta em cloudflare.com (Free plan)
  2. Clique em Add a site → digite seu domínio
  3. Escolha o plano Free
  4. O Cloudflare vai escanear seus DNS records
  5. Adicione o CNAME para Vercel: CNAME → cname.vercel-dns.com
  6. No seu registrador de domínio, mude os nameservers para os do Cloudflare
  7. Aguarde propagação (minutos a horas)

09 Manutenção e Interoperabilidade

Ferramentas de segurança só funcionam se você mantê-las atualizadas. Aqui está o que precisa de manutenção.

🔄 9.1 Dependências — Atualização contínua

O Dependabot abre PRs automaticamente no GitHub. Seu trabalho:

  1. Toda segunda-feira, verifique se há PRs do Dependabot
  2. Para atualizações de patch (1.2.3 → 1.2.4): aprove e merge
  3. Para atualizações de minor (1.2.x → 1.3.x): teste antes no preview
  4. Para atualizações de major (1.x → 2.x): pesquise breaking changes antes
🔑 9.2 Rotação de chaves

A cada 3 meses, rotacione as chaves do Supabase:

  1. Supabase → Settings → API → Regenerate anon key
  2. Atualize na Vercel: Settings → Environment Variables
  3. Atualize no Lovable (se necessário)
  4. Redeploy
  5. Teste que tudo funciona
⚠️ A anon key antiga para de funcionar imediatamente. Faça isso em horário de baixo uso.

10 Roteiro de Monitoramento Contínuo
📅 Rotina Diária (2 minutos)
  • Abrir email → verificar alertas do Sentry e UptimeRobot
  • Se houver alerta → investigar imediatamente
📅 Rotina Semanal (10 minutos)
  • GitHub → Actions → verificar ✅ ou ❌ nos scans
  • GitHub → Security → verificar alertas do Dependabot
  • GitHub → Pull Requests → merge PRs do Dependabot (patch/minor)
  • Sentry → verificar erros novos e tendências
  • Supabase → SQL Editor → query de deleções recentes:
    SELECT * FROM audit_log WHERE action='DELETE' AND timestamp > now()-interval '7 days'
📅 Rotina Mensal (30 minutos)
  • Rodar pipeline SAST manualmente (GitHub → Actions → Run workflow)
  • Baixar e revisar relatório completo
  • Supabase → Table Editor → verificar RLS em TODAS as tabelas
  • Supabase → Auth → verificar usuários (algum suspeito?)
  • Cloudflare → Analytics → verificar tráfego bloqueado
  • Cloudflare → Security → WAF → verificar eventos
  • Atualizar /docs/SECURITY_REPORT.md
📅 Rotina Trimestral (1 hora)
  • Rotacionar chaves do Supabase
  • Revisar TODAS as RLS policies em detalhe
  • Executar testes DAST completos (Capítulo 03)
  • Revisar dependências — alguma deprecated?
  • Verificar se há atualizações nas regras do Semgrep
  • Atualizar SECURITY_RULES.md se necessário
11 Ferramentas que Você Não Pensou

Além de segurança pura, há aspectos que muita gente esquece e que impactam diretamente a saúde do projeto.

🔐 Bitwarden / 1Password

Gerenciador de senhas para equipe. Se você tem clientes e projetos múltiplos, precisa guardar chaves de API, credenciais do Supabase, tokens de serviços — tudo em um lugar seguro, não em arquivos de texto ou WhatsApp.
Bitwarden: gratuito para uso pessoal Compartilhamento seguro de credenciais

📋 Plausible / Umami Analytics

Analytics sem cookies e GDPR-compliant. Substitui o Google Analytics. Não rastreia dados pessoais, é leve, e você consegue ver o tráfego real. Útil para detectar picos anormais (possível ataque).
Umami: self-hosted gratuito Plausible: ~$9/mês GDPR compliant

📧 Resend + React Email

Emails transacionais seguros. Confirmação de email, reset de senha, notificações — com domínio customizado e DKIM/SPF configurados para evitar que caiam no spam ou sejam falsificados.
Resend: 3K emails/mês grátis DKIM + SPF = anti-phishing

🗂️ Supabase Vault (criptografia)

Criptografia de dados sensíveis no banco. Se você armazena CPF, dados de saúde ou financeiros, o Vault do Supabase permite criptografar colunas específicas. Mesmo que alguém acesse o banco, os dados ficam ilegíveis.
Supabase nativo Ideal para dados de saúde/financeiros

🧪 Checkly / Playwright

Testes end-to-end automatizados. Simula um usuário real navegando no app: faz login, clica em botões, preenche formulários. Se algo quebrar, você sabe antes do usuário.
Checkly: 50 checks grátis Playwright: open-source

📜 Termos de Uso e Política de Privacidade

Aspecto legal que muita gente esquece. Se sua aplicação coleta dados pessoais (e se usa Supabase Auth, ela coleta), você PRECISA de: Termos de Uso, Política de Privacidade, e em alguns casos Cookie Banner. Não é paranoia — é LGPD (no Brasil) e GDPR (na Europa).
Obrigatório por lei (LGPD) Pode gerar com IA e revisar com advogado

🔄 Supabase Branching

Ambientes separados para dev/staging/produção. Evita que testes em desenvolvimento afetem dados reais. O Lovable trabalha com uma instância — para produção, considere ter ambiente separado.
Supabase Pro Alternativa: criar um segundo projeto gratuito para dev
