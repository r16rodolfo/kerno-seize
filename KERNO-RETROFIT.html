Kerno Retrofit

Processo passo a passo para adotar o framework Kerno Seize em projetos Lovable que já existem — sem quebrar o que funciona.

📍 Seu cenário

Você tem projetos Lovable conectados ao Supabase e Vercel, no ar mas só você usa/testa, sem documentação estruturada. Precisa aplicar segurança, hardening de deploy, estabilidade e documentação — nessa ordem de prioridade.

❌ Abordagem errada

Tentar fazer tudo de uma vez. Pedir ao Lovable para "refatorar todo o projeto com segurança e documentação". Resultado: ele vai quebrar coisas, misturar mudanças, e você não vai saber o que voltou a funcionar.

✅ Abordagem correta

Fase por fase, começando pelo que dá mais risco. Fixar versão antes de cada fase. Verificar que tudo funciona antes de avançar. Uma coisa por vez — exatamente como a documentação do Lovable recomenda.

🧭 Princípio deste framework: você não precisa saber o que está fazendo tecnicamente

Cada passo deste guia foi desenhado para que você não precise tomar decisões técnicas. O Lovable analisa, classifica e te diz o que está errado — e em qual ordem corrigir. Quando o resultado for confuso, cada passo indica exatamente o que copiar e onde colar para obter ajuda. Siga os passos na ordem. Não pule etapas. Não improvise.

🔀 Antes de tudo: Retrofit ou Reconstrução?

Nem todo projeto existente vale a pena recuperar. Use o diagnóstico abaixo para obter um veredicto automático antes de gastar tempo no processo de Retrofit — o próprio Lovable faz a análise e calcula a pontuação.

Cole este prompt no Lovable (Plan Mode, sem editar código):

MODO LEITURA APENAS. Nenhum arquivo deve ser criado, modificado ou removido. Esta é uma análise de diagnóstico — apenas leitura e resposta em texto. Leia o código e os arquivos do projeto e gere um relatório de diagnóstico com o seguinte formato: --- DIAGNÓSTICO KERNO RETROFIT --- PARTE 1 — PONTUAÇÃO DE SAÚDE (máximo 70 pontos) Para cada item abaixo, informe se a condição é verdadeira ou falsa no projeto atual e a pontuação correspondente: Item S1: O schema do banco reflete o produto real, sem tabelas de teste ou colunas sem uso. Verdadeiro: 15 pontos. Falso: 0 pontos. Item S2: Autenticação existe e o fluxo principal de login/logout funciona. Verdadeiro: 15 pontos. Falso: 0 pontos. Item S3: Há no máximo 2 tabelas sem RLS ativo. Verdadeiro: 10 pontos. Falso: 0 pontos. Item S4: A chave service_role está ausente do frontend, ou aparece em no máximo 2 arquivos. Verdadeiro: 10 pontos. Falso: 0 pontos. Item S5: O fluxo principal do produto funciona de ponta a ponta sem erros críticos. Verdadeiro: 10 pontos. Falso: 0 pontos. Item S6: O projeto tem menos de 60 dias de desenvolvimento acumulado. Verdadeiro: 5 pontos. Falso: 0 pontos. Item S7: As funcionalidades do projeto são descritíveis sem necessidade de ler o código. Verdadeiro: 5 pontos. Falso: 0 pontos. PARTE 2 — PONTUAÇÃO DE RISCO (desconta pontos) Para cada item abaixo, informe se a condição é verdadeira ou falsa: Item R1: Há mais de 4 tabelas sem RLS contendo dados reais de usuários. Verdadeiro: desconta 20 pontos. Falso: desconta 0 pontos. Item R2: A chave service_role aparece em 3 ou mais componentes do frontend. Verdadeiro: desconta 20 pontos. Falso: desconta 0 pontos. Item R3: O schema está irreconhecível, com colunas duplicadas ou migrations quebradas. Verdadeiro: desconta 15 pontos. Falso: desconta 0 pontos. Item R4: O produto nunca funcionou de forma estável — há mais bugs do que funcionalidades. Verdadeiro: desconta 15 pontos. Falso: desconta 0 pontos. Item R5: O histórico de sessões mostra código inconsistente por perda de contexto. Verdadeiro: desconta 10 pontos. Falso: desconta 0 pontos. Item R6: A dívida técnica aparente é superior a 7 em uma escala de 0 a 10. Verdadeiro: desconta 10 pontos. Falso: desconta 0 pontos. PARTE 3 — RESULTADO FINAL Informe: - Subtotal de saúde (soma dos itens S1 a S7) - Subtotal de risco (soma dos descontos R1 a R6) - Pontuação final (subtotal de saúde menos subtotal de risco) - Os 2 ou 3 fatores que mais influenciaram o resultado PARTE 4 — VEREDICTO Com base na pontuação final, emita exatamente um dos três veredictos abaixo: Pontuação 40 ou mais: RETROFIT VIAVEL Pontuação de 20 a 39: RETROFIT ARRISCADO Pontuação abaixo de 20: RECONSTRUCAO RECOMENDADA LEMBRETE FINAL: Este prompt é somente de análise. Nenhum código foi alterado.

✅ Retrofit viável — quando:

  • Schema do banco reflete o produto real (não tem tabelas "de teste" aleatórias)
  • Autenticação já existe e funciona
  • Máximo 1–2 tabelas sem RLS (corrigível em <30 min)
  • Não há service_role no frontend, ou há em apenas 1–2 lugares
  • O produto está funcional — usuário consegue fazer o fluxo principal
  • Menos de 60 dias de desenvolvimento acumulado (base ainda não explodiu)
  • Você entende o que o código faz (consegue descrever as features para o Lovable)
👉 Siga o Retrofit. As fases abaixo resolvem os problemas sem recriar o que funciona.

🔁 Reconstrução recomendada — quando:

  • Schema está irreconhecível — tabelas sem sentido, colunas duplicadas, migrations quebradas
  • Mais de 4 tabelas sem RLS com dados reais de usuário
  • service_role espalhado por múltiplos componentes no frontend
  • O produto nunca chegou a funcionar direito — mais bugs do que features
  • O Lovable perdeu o contexto do projeto em múltiplas sessões (código inconsistente)
  • Você não consegue descrever o que o código faz sem lê-lo linha a linha
  • Dívida técnica > 7/10 na avaliação acima
⚠️ Não faça Retrofit. O custo de consertar será maior do que reconstruir com o BMAD do zero. Use o KERNO-ROADMAP.html (Etapa 0 em diante).

⚖️ Casos intermediários — o que fazer

Situação Decisão Justificativa
Schema OK, mas UI completamente errada Retrofit parcial Preserve o banco e reconstrói só o frontend com WDS Onda 2
Produto funciona, mas escopo mudou muito Depende Se >60% das features mudam, reconstrução com BMAD é mais rápida a longo prazo
Múltiplos problemas de segurança críticos Retrofit urgente Corrija segurança primeiro (Fases 1–2). Considere reconstrução depois
Projeto com usuários reais pagantes Retrofit Não pode tirar do ar. Faça o Retrofit com muito cuidado na Fase 0

FASE 2

Hardening de Deploy

⏱️ 20-40 min por projeto

Mover o projeto para uma infraestrutura de produção segura — com headers de segurança, variáveis de ambiente protegidas e um deploy profissional fora do preview do Lovable.

🔍 2.0 Primeiro: descubra onde seu projeto está hoje

Antes de fazer qualquer coisa, você precisa saber em qual situação está. Abra o Lovable no projeto e olhe o topo da tela — onde fica o botão de publicar/deploy.

SITUAÇÃO A
Só tenho o preview do Lovable
A URL do projeto termina em .lovable.app ou similar. Nunca foi publicado em outro lugar. Não sei o que é Vercel.
→ Siga o Caminho A abaixo
SITUAÇÃO B
Já está na Vercel (ou Netlify)
Tenho uma conta na Vercel/Netlify com este projeto conectado. A URL vem de lá, ou eu mesmo conectei o GitHub.
→ Siga o Caminho B abaixo
💡 Não sabe qual situação é a sua? Se o seu app está em meu-app.lovable.app ou meu-app.lovableproject.com, você está na Situação A. Se a URL tem o nome da Vercel ou um domínio próprio que você configurou manualmente, é a Situação B.
▼ CAMINHO A — Saindo do preview Lovable e indo para a Vercel
A.1 Conectar o projeto ao GitHub

No Lovable, clique no ícone do GitHub (canto superior direito do editor) e escolha "Connect to GitHub". Siga o fluxo de autorização. O Lovable vai criar automaticamente um repositório com o código do seu projeto.

💡 Isso não publica nada ainda. Só cria uma cópia do código no GitHub — que é necessária para o próximo passo.
A.2 Preparar o arquivo de segurança antes de publicar

Com o GitHub conectado, envie este prompt no Lovable antes de conectar a Vercel:

Crie o arquivo vercel.json na raiz do projeto com headers de segurança completos:
X-Frame-Options DENY, X-Content-Type-Options nosniff,
Referrer-Policy strict-origin-when-cross-origin,
Permissions-Policy restritiva, HSTS com max-age 2 anos,
e Content-Security-Policy que permita apenas self, Supabase e fontes do Google.
Não altere nenhum outro arquivo.
Confirme com: "✅ vercel.json criado"
A.3 Importar na Vercel
  1. Acesse vercel.com — crie uma conta gratuita com o mesmo email do GitHub
  2. Clique em "Add New Project"
  3. Escolha "Import Git Repository" e selecione o repositório que o Lovable criou
  4. Na tela de configuração, não mexa em nada — a Vercel detecta o projeto automaticamente
  5. Antes de clicar em Deploy, expanda "Environment Variables" e adicione:
    • VITE_SUPABASE_URL — valor: a URL do seu projeto no Supabase
    • VITE_SUPABASE_ANON_KEY — valor: a chave anon/public do Supabase
  6. Clique em "Deploy"
💡 Onde encontrar os valores do Supabase: acesse app.supabase.com → seu projeto → Settings → API. Os dois valores estão lá, claramente identificados.
A.4 Confirmar que está funcionando

A Vercel vai gerar uma URL como meu-app.vercel.app. Abra essa URL e teste: o app carrega? Login funciona? Se sim, você tem dois deploys funcionando — o Lovable Cloud (original) e a Vercel (novo). A partir de agora, use a URL da Vercel.

⚠️ O Lovable Cloud continua funcionando normalmente. Não precisa (nem deve) excluir nada. Ele serve como backup enquanto você valida que a Vercel está tudo certo.
▼ CAMINHO B — Projeto já está na Vercel, adicionar hardening
B.1 Verificar variáveis de ambiente

Na Vercel, abra o projeto → Settings → Environment Variables. Verifique se existem as duas variáveis abaixo apenas com o escopo "Production" selecionado (não "Preview" ou "Development"):

  • VITE_SUPABASE_URL
  • VITE_SUPABASE_ANON_KEY
🚨 Se não existem ou estão com escopo errado: adicione/corrija agora. Sem essas variáveis, o Supabase não funciona na Vercel e você pode estar usando valores hardcoded no código — o que é um risco de segurança.
B.2 Adicionar headers de segurança

Envie este prompt no Lovable:

Verifique se existe um arquivo vercel.json na raiz do projeto.
Se não existir, crie com headers de segurança completos:
X-Frame-Options DENY, X-Content-Type-Options nosniff,
Referrer-Policy strict-origin-when-cross-origin,
Permissions-Policy restritiva, HSTS com max-age 2 anos,
e Content-Security-Policy que permita apenas self, Supabase e fontes do Google.
Se já existir, verifique se tem esses headers e adicione os que estiverem faltando.
Não altere nenhum outro arquivo.
Confirme com: "✅ vercel.json atualizado" ou "✅ vercel.json já estava correto"
🗄️ 2.1 Atualizar URLs no Supabase (para todos)

Agora que o projeto tem uma URL de produção, o Supabase precisa saber qual é essa URL para que o login e os redirecionamentos funcionem corretamente. Envie este prompt no Lovable:

NÃO altere nenhum código. Me diga apenas:
1. Qual é a URL de produção atual deste projeto (vercel.app ou domínio próprio)?
2. Quais URLs de redirecionamento de autenticação estão configuradas no Supabase?
3. Elas incluem a URL de produção atual?

Se não incluírem, me dê as instruções exatas de onde clicar no painel do Supabase
para adicionar a URL correta — em linguagem simples, passo a passo.
💡 O Lovable vai te dar as instruções exatas para o painel do Supabase, personalizadas para o seu projeto. Siga o que ele disser.
🔐 2.2 Ativar proteções no GitHub (2 minutos)

No GitHub, abra o repositório do projeto → Settings → Code security and analysis. Ative os quatro botões abaixo clicando em "Enable" em cada um:

  • Secret scanning — detecta chaves e senhas acidentalmente commitadas
  • Push protection — bloqueia pushes com secrets antes de chegarem ao repositório
  • Dependabot alerts — avisa quando uma dependência tem vulnerabilidade conhecida
  • Dependabot security updates — abre PRs automáticos para corrigir dependências vulneráveis
💡 Cada um tem um botão "Enable" ao lado. São apenas cliques — sem configuração adicional.
✅ 2.3 Validar e fechar a fase

Peça ao Lovable para confirmar que tudo está em ordem:

NÃO altere nenhum código. Verifique e me diga:
1. O arquivo vercel.json existe e tem headers de segurança?
2. As variáveis de ambiente VITE_SUPABASE_URL e VITE_SUPABASE_ANON_KEY
   estão no código ou referenciadas via import.meta.env?
3. Há algum outro problema de configuração de deploy que devo resolver?

Responda de forma simples: OK ou PROBLEMA ENCONTRADO para cada item.

Depois teste manualmente: abra a URL de produção, faça login, navegue pelas telas principais. Se tudo funciona → 📌 Fixe a versão no Lovable: "Deploy hardening completo — [DATA]"

FASE 3

Estabilidade

⏱️ 15-20 min por projeto

Configurar o Lovable para parar de quebrar coisas quando evolui o projeto.

🧠 3.1 Custom Knowledge já está configurado

Se você fez a Fase 0.3, as regras de estabilidade já estão no Custom Knowledge:

  • Prompts incrementais (uma coisa por vez)
  • Nunca alterar componentes não relacionados
  • Planejar antes de implementar (Plan Mode)
  • Recomendar fixar versão após cada feature
  • Alertar antes de mudanças em schema do Supabase
📐 3.2 Adotar o padrão de prompts

A partir de agora, sempre use este formato ao pedir features:

[FEATURE] Descrição clara e específica do que quero.

Regras:
- Implemente APENAS isto, não mexa em nada que não seja
  diretamente necessário
- Se precisar alterar algo existente, me diga ANTES
- Atualize os /docs correspondentes
- Me diga se devo fixar a versão atual antes desta mudança
🔄 3.3 Fluxo de trabalho estável

Adote este ciclo para cada sessão de desenvolvimento:

  1. Verificar — o app funciona no preview? Sim → continuar
  2. Fixar — pin na versão atual antes de começar
  3. Planejar — Plan Mode para features complexas
  4. Implementar — um pedido por vez
  5. Testar — funciona? Login, CRUD, responsivo
  6. Fixar — pin se funciona, revert se não
  7. Fechar — prompt de fechamento de sessão

FASE 4.5

Auditoria BMAD — Revisão Inteligente dos Documentos

⏱️ 1h30–3h · Mensal

Após gerar a documentação retroativa na Fase 4, os agentes BMAD entram em modo de auditoria: revisam tudo que o Lovable gerou e produzem um relatório de insights, correções e melhorias — antes de escalar.

💡 Dica v5.5 — Context Scanner: Se o AI PM Copilot estiver instalado, rode /ai-pm-copilot:pm-setup antes da auditoria. O context-scanner lê o codebase automaticamente e gera um mapa de features, stack e integrações. Isso alimenta Winston, Bob, Mary e John com dados reais em vez de descrição manual. Ver KERNO-PM-INTEGRACAO.md
⚠️ Por que esta fase é fundamental: O Lovable gera documentação e código funcionais, mas não faz revisão crítica. Sem auditoria, você vai escalar um projeto com código redundante, jornadas de UX quebradas, documentação inconsistente e dívida técnica acumulada.
WINSTON
Arquitetura
Arquivos duplicados, componentes mortos, estrutura que não escala
BOB
Qualidade de Código
Código verboso, lógica repetida, padrões inconsistentes
MARY
Produto + UX
Jornadas quebradas, gaps de UX, PRD inconsistente com o real
JOHN
Drift Analysis
Divergências entre documentação e código real
🔁 Governança mensal: a auditoria usa pastas datadas (/docs/audit/YYYY-MM-DD/), histórico em AUDIT-HISTORY.md, e comparação automática com a rodada anterior para rastrear evolução e itens recorrentes.
Ver módulo completo → KERNO-SEIZE-AUDITORIA.html

🤖 E os agentes BMAD + PM Skills?

No retrofit, as ferramentas atuam em três momentos distintos:

  1. Retroativo (uma vez): PM Skills define métricas (/north-star), mapeia concorrência (/competitive-analysis), e articula trade-offs (/strategy). AI PM Copilot context-scanner gera mapa do codebase. Esses inputs alimentam a auditoria BMAD.
  2. Fase 4.5 — Auditoria: Winston, Bob, Mary e John revisam o que o Lovable gerou e produzem relatórios de melhoria (ver módulo KERNO-SEIZE-AUDITORIA.html)
  3. Expansão futura: ao adicionar funcionalidades novas, use PM Skills (/discover, /value-proposition) para validar antes, depois John para PRD, Winston para arquitetura, e envie os documentos ao Lovable como input

O fluxo completo para projetos novos está no KERNO-ROADMAP.html. Ver também KERNO-PM-INTEGRACAO.md para o pipeline PM Skills + BMAD.
