Auditoria BMAD

Use os agentes do BMAD para revisar, corrigir e elevar a qualidade de tudo que o Lovable gerou — antes de escalar.

Execute após gerar as documentações pelo Lovable

🚨 Por que este módulo existe?

O Lovable é uma ferramenta poderosa de geração, mas não de revisão crítica. Quando você começa um projeto sem planejamento — como a maioria das pessoas faz — o Lovable vai implementar o que você pediu da forma que entendeu, e muitas vezes isso significa:

❌ O que o Lovable faz sem supervisão

  • Cria 3 componentes onde bastaria 1
  • Escreve 200 linhas de código que poderiam ser 40
  • Gera arquivos de configuração que nunca são usados
  • Duplica lógica em múltiplos lugares
  • Cria jornadas de UX quebradas ou confusas
  • Documenta o que fez, não o que deveria ser
  • Perde contexto entre sessões e cria inconsistências

✅ O que a Auditoria BMAD entrega

  • Relatório de código redundante e como eliminar
  • Mapa das jornadas de UX com falhas identificadas
  • Documentação corrigida e consistente com o código real
  • Lista priorizada de refatorações por impacto
  • Arquitetura limpa antes de escalar
  • Insights de melhorias que o Lovable não vê
  • Base sólida para o próximo ciclo de desenvolvimento
⚠️ Escalar código ruim custa 10x mais para corrigir depois. Um projeto com 500 linhas problemáticas vira 5.000 linhas problemáticas em 3 meses se você não auditar antes de crescer.

@bob modo análise + geração de relatório. Não modifique nenhum arquivo existente.

Faça uma revisão de qualidade do código deste projeto como um dev sênior experiente.

PARTE 1 — CÓDIGO VERBOSO
- Funções com mais de 50 linhas que poderiam ser menores
- Blocos de código que se repetem em 2+ lugares (candidatos a extração)
- Condicionais aninhados que poderiam ser simplificados
- Código comentado que nunca foi removido

PARTE 2 — PADRÕES INCONSISTENTES
- Mistura de estilos (arrow functions vs function declarations, async/await vs .then())
- Nomes de variáveis/funções inconsistentes ou mal nomeados
- TypeScript mal usado (any excessivo, tipagens faltando onde importa)
- Import paths inconsistentes (relativos vs absolutos)

PARTE 3 — LÓGICA PROBLEMÁTICA
- useEffect com dependências erradas ou faltando
- Estado sendo duplicado em múltiplos componentes que poderiam compartilhá-lo
- Chamadas de API sem tratamento de erro
- Dados sendo buscados em loop ou de forma ineficiente

PARTE 4 — OPORTUNIDADES DE ABSTRAÇÃO
- Lógica repetida que deveria virar um custom hook
- Componentes UI genéricos que estão sendo reescritos ao invés de reutilizados
- Utilitários que estão inline quando deveriam ser funções isoladas

PARTE 5 — RANKING DE CORREÇÕES
- Liste as 10 correções mais impactantes em ordem de prioridade
- Para cada uma: arquivo específico, problema, solução sugerida, esforço estimado

Seja específico: cite arquivos e linhas quando possível.

PARTE 1 — DOCS vs. CÓDIGO

Drift entre documentação e código é silencioso e perigoso. Quando um novo dev (ou o Lovable numa nova sessão) lê a documentação e implementa baseado nela, vai criar bugs que são difíceis de rastrear porque "segui o doc".
