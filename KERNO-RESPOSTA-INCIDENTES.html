Guia Operacional

Resposta a Incidentes

O que fazer quando algo dá errado. Playbooks passo a passo para cada tipo de incidente — do "app caiu" ao "fomos invadidos".

00 Princípios de Resposta

Antes dos playbooks, internalize estes princípios. Eles se aplicam a QUALQUER incidente.

1. Não entre em pânico

Decisões apressadas causam mais dano que o incidente original. Respire, leia o playbook, siga os passos.

2. Contenha ANTES de corrigir

Primeiro pare o sangramento (bloquear acesso, reverter deploy). Depois investigue a causa. Corrigir sem conter pode piorar.

3. Documente tudo

Anote: hora que descobriu, o que viu, o que fez, o resultado. Isso é essencial para aprender e evitar que aconteça de novo.

4. Preserve evidências

Antes de deletar ou corrigir, tire screenshots, exporte logs, copie dados. Se for invasão, você pode precisar dessas evidências.

5. Comunicação

Se o incidente afeta clientes, comunique rapidamente: "Identificamos um problema, estamos trabalhando na solução". Silêncio é pior que más notícias.

02 Playbooks por Incidente
CRÍTICO App fora do ar (UptimeRobot alertou) Meta: resolver em 30 min

Você recebeu: Email do UptimeRobot dizendo "Monitor is DOWN"

1
Confirme manualmente. Abra a URL no navegador e no celular. Pode ser falso positivo do UptimeRobot.
2
Se confirmou que está fora, verifique a Vercel: Abra vercel.com → seu projeto → Deployments. O último deploy está com ❌? Se sim, o problema é no deploy.
3
Se deploy falhou: Clique no deploy anterior que estava ✅ → botão "Promote to Production". Isso restaura a versão que funcionava em segundos.
4
Se deploy está ✅ mas app não carrega: Verifique o Supabase. Abra supabase.com → seu projeto. Se o dashboard não abrir ou mostrar erro, o problema é no Supabase (raro, geralmente é manutenção deles).
5
Se Supabase está ok: Verifique o Cloudflare (se usa). Security → Events. Pode estar bloqueando tráfego legítimo. Temporariamente mude Security Level para "Essentially Off".
6
Se nada acima funcionou: Abra o Sentry para ver se há erros. Abra o console do navegador (F12) na URL de produção para ver erros JavaScript.
7
Após resolver: Documente o que aconteceu no CHANGELOG.md. Verifique que o UptimeRobot mudou para "UP".
💡 90% das vezes um app Lovable+Vercel cai por: deploy com erro (resolve revertendo) ou Supabase fora (aguardar). Raramente é algo complexo.
ALTO Pico de erros no Sentry Meta: investigar em 4h

Você recebeu: Alerta do Sentry — "Event frequency exceeded threshold"

1
Abra o Sentry → Issues. Veja qual erro está disparando. Clique nele para ver o stack trace (onde no código o erro acontece).
2
Verifique o padrão: É um erro novo? Começou após algum deploy? Se sim → o deploy introduziu o bug. Reverta na Vercel (Promote deploy anterior).
3
Verifique os usuários: No Sentry, veja se o erro vem de muitos usuários ou de um só. Um único usuário com centenas de erros pode indicar bot ou tentativa de ataque.
4
Se parece ataque: Verifique o Cloudflare Security → Events. Se houver IPs suspeitos, bloqueie-os. Verifique audit_log no Supabase para esse período.
5
Se é bug legítimo: Corrija no Lovable com prompt específico: "Corrija o erro [descrição] no arquivo [arquivo]. NÃO altere mais nada." Pin → Fix → Test → Pin.
6
Após resolver: No Sentry, marque o issue como "Resolved". Se voltar a acontecer, o Sentry reabrirá automaticamente.
CRÍTICO Secret vazou (chave de API no código) Meta: resolver IMEDIATAMENTE

Você recebeu: Alerta do GitHub Secret Scanning, OU percebeu uma chave no código

🚨 Este é o incidente mais urgente. Uma chave exposta pode ser usada em minutos por bots que escaneiam o GitHub continuamente.
1
ROTACIONE A CHAVE IMEDIATAMENTE. Vá ao serviço afetado (Supabase, Sentry, etc.) e gere uma nova chave. A chave antiga morre na hora.
2
Atualize a nova chave na Vercel: Settings → Environment Variables → substitua o valor → Redeploy.
3
Verifique se foi usada: No Supabase, verifique audit_log das últimas horas. Houve ações que você não fez? No Auth, tem usuários estranhos?
4
Remova do código: No Lovable, peça: "Remova qualquer chave hardcoded e use import.meta.env.VITE_* em vez disso."
5
Limpe o histórico: Mesmo depois de remover do código, a chave AINDA ESTÁ no histórico do Git. Se o repo é público, considere torná-lo privado. Para repos privados, o risco é menor mas a chave já foi rotacionada no passo 1.
⚠️ Se a chave era a service_role do Supabase: isso é equivalente a root access ao banco. Verifique TODOS os dados. Se houver dados de clientes, pode ser necessário notificá-los (LGPD Art. 48).
CRÍTICO Atividade suspeita / possível invasão Meta: conter em 1h

Sinais: Deleções em massa no audit_log, usuários desconhecidos, dados alterados, acesso fora do horário

1
Contenha: Se identificou o usuário suspeito → Supabase → Auth → selecione o usuário → "Ban User". Isso bloqueia o acesso imediatamente.
2
Preserve evidências: Execute AGORA no Supabase SQL Editor: 
-- Exportar audit_log das últimas 48h
SELECT * FROM audit_log
WHERE timestamp > now() - interval '48 hours'
ORDER BY timestamp DESC;
Copie o resultado e salve em arquivo separado.
3
Analise a extensão: Quais tabelas foram afetadas? Quais dados foram acessados/alterados? Use: 
-- O que o invasor fez?
SELECT action, table_name, count(*)
FROM audit_log
WHERE user_id = 'UUID_SUSPEITO'
AND timestamp > now() - interval '48 hours'
GROUP BY action, table_name;
4
Rotacione TODAS as chaves: anon key, service_role (se comprometida), qualquer API key que o invasor possa ter visto.
5
Verifique RLS: O invasor conseguiu acessar dados que não deveria? Se sim, suas RLS policies têm falhas. Corrija ANTES de reabrir o acesso.
6
Se dados pessoais foram comprometidos (LGPD): Você tem obrigação legal de notificar a ANPD (Autoridade Nacional de Proteção de Dados) e os titulares dos dados em prazo razoável. Consulte um advogado.
7
Pós-incidente: Documente tudo. Faça revisão completa de RLS. Considere: como o invasor entrou? Senha fraca? RLS faltando? Secret exposto? Corrija a causa raiz.
MÉDIO Alerta do Dependabot (CVE em dependência) Meta: avaliar em 24h, resolver em 1 semana

Você recebeu: Email/PR do Dependabot com vulnerabilidade

1
Leia a severidade: GitHub → Security → Dependabot alerts. Veja se é Critical, High, Medium ou Low.
2
Verifique se afeta você: Muitas CVEs são em funcionalidades que você não usa. Leia a descrição. Se é uma vulnerabilidade em "server-side XML parsing" e você não processa XML, o risco real é baixo.
3
Se o Dependabot abriu PR automático: Verifique se a atualização é patch (1.2.3→1.2.4). Se sim, aprove e merge. O pipeline SAST vai rodar automaticamente.
4
Se é major update (1.x→2.x): Teste no preview da Vercel antes de fazer merge. Major updates podem ter breaking changes.
5
Se não há fix disponível: Avalie se a dependência pode ser substituída. Marque o alert como "Dismissed" com motivo se decidir não corrigir.
ALTO Pipeline SAST falhou (Semgrep encontrou algo) Meta: avaliar em 4h
1
Abra GitHub → Actions → clique na execução com ❌. Veja qual job falhou (Semgrep, npm audit, ou Scanner).
2
Baixe o relatório em Artifacts. Abra o HTML para ver os detalhes com arquivo, linha e tipo do problema.
3
Classifique: É um problema real ou falso positivo? Semgrep às vezes alerta sobre padrões que no seu contexto são seguros.
4
Se é real: Corrija no Lovable com prompt específico citando o arquivo e a linha. Teste. O pipeline vai rodar novamente automaticamente no push.
5
Se é falso positivo: Adicione um comentário // nosemgrep: rule-id na linha para silenciar.
ALTO Deleções incomuns no audit_log Meta: investigar em 4h

Você encontrou: Na revisão semanal, várias deleções que você não fez

1
Identifique o usuário: 
SELECT user_id, table_name, count(*), min(timestamp), max(timestamp)
FROM audit_log
WHERE action = 'DELETE'
AND timestamp > now() - interval '7 days'
GROUP BY user_id, table_name
ORDER BY count DESC;
2
Veja os detalhes: O que foi deletado? 
SELECT * FROM audit_log
WHERE action = 'DELETE'
AND user_id = 'UUID_DO_USUARIO'
ORDER BY timestamp DESC LIMIT 20;
3
Avalie: O usuário tinha permissão para deletar? Era uso normal do app (cliente deletando seus próprios dados) ou anômalo (alguém deletando dados de outros)?
4
Se anômalo: Siga o playbook de "Atividade suspeita / possível invasão" acima.
5
Recuperação: Os dados antigos estão no campo old_data do audit_log. Você pode reconstruir registros deletados a partir dessa informação.
CRÍTICO Tabela sem RLS descoberta Meta: corrigir IMEDIATAMENTE
1
Verifique AGORA: No Supabase SQL Editor: 
-- Listar tabelas sem RLS
SELECT schemaname, tablename, rowsecurity
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY tablename;
Se rowsecurity está false, a tabela está exposta.
2
Ative RLS imediatamente: ALTER TABLE nome_tabela ENABLE ROW LEVEL SECURITY;
3
CUIDADO: Ativar RLS sem policies bloqueia TODO acesso (inclusive legítimo). Crie policies básicas: 
-- Policy básica: usuários veem apenas seus dados
CREATE POLICY "Users see own data" ON nome_tabela
  FOR SELECT USING (auth.uid() = user_id);

-- Admins veem tudo
CREATE POLICY "Admins full access" ON nome_tabela
  FOR ALL USING (
    EXISTS (SELECT 1 FROM user_roles
    WHERE user_roles.user_id = auth.uid()
    AND role = 'admin')
  );
4
Teste: Faça login como usuário normal e verifique que ainda consegue usar o app normalmente.
5
Verifique no audit_log: Alguém acessou dados dessa tabela indevidamente antes de você corrigir?
03 Estratégia de Monitoramento com Ferramentas Prontas

Em vez de construir um painel do zero, use as ferramentas gratuitas de forma inteligente. Uma conta de cada resolve até 6 projetos.

FerramentaPlano FreeSuficiente paraO que monitorar
Sentry5K erros/mês, projetos ilimitados, 1 usuário4-6 apps pequenasErros JS, crashes, performance
UptimeRobot50 monitores, check a cada 5min~16 projetos (3 monitores/projeto)Uptime, response time, SSL
GitHubActions, Dependabot, Secret scan ilimitadosQualquer quantidadeSAST, CVEs, secrets
Cloudflare1 site/conta (criar conta por domínio)Ilimitado (1 conta free por domínio)WAF, DDoS, CDN, analytics
Supabase2 projetos free / contaCriar conta por par de projetosAuth, banco, audit_log
💡 Custo total: R$ 0. Uma conta Sentry + uma conta UptimeRobot + GitHub + Cloudflare Free + Supabase Free cobrem uma agência com 4-6 projetos sem pagar nada em dólar.

Como organizar na prática

1
Sentry: Uma conta, um projeto por app. O dashboard do Sentry já mostra todos os projetos lado a lado. Favoritar os mais importantes.
2
UptimeRobot: Uma conta, 3 monitores por app (frontend, API, login). O dashboard mostra todos juntos. Configurar alertas por email E pelo app mobile.
3
GitHub: Uma aba do navegador fixa em github.com/notifications. Todas as alertas de segurança de todos os repos aparecem aqui.
4
Abas fixas no navegador: Fixe 3 abas (Sentry, UptimeRobot, GitHub). Isso funciona como um "mini painel" sem construir nada.
5
Audit logs: Este é o ÚNICO item que não tem ferramenta pronta. Consultar semanalmente via Supabase SQL Editor em cada projeto.
04 Pós-Incidente — Aprendizado

Depois de resolver qualquer incidente MÉDIO ou acima, faça uma mini retrospectiva. Não precisa ser formal — 10 minutos anotando:

Template — Relatório pós-incidente
## Incidente: [título curto]
**Data:** YYYY-MM-DD HH:MM
**Severidade:** Crítico / Alto / Médio
**App afetado:** [nome]
**Duração:** [quanto tempo durou]

### O que aconteceu
[Descreva em 2-3 frases]

### Como descobrimos
[UptimeRobot alertou? Sentry? Usuário reportou? Revisão semanal?]

### O que fizemos
[Passos que seguiu para resolver]

### Causa raiz
[Por que aconteceu? Deploy sem testar? RLS faltando? Dependência?]

### O que vamos fazer para não repetir
[Ação concreta: adicionar teste, corrigir RLS, mudar processo]
💡 Salve esses relatórios no /docs/INCIDENT_LOG.md do projeto. Com o tempo, você terá um histórico que mostra padrões e melhora seus processos.
